Auth & the provider model

A design recommendation for how a github://owner/repo provider — and every future auth-requiring provider (ADO work items, Planner, …) — logs a user in, holds their credentials, and drives authenticated network calls, without breaking the separation of concerns between the engine, a provider, and a host.

recommendation — not yet implemented companion to architecture.html stress-tested via two adversarial reviews — see risk register
The one-sentence version: the engine never learns what a token is; a provider only ever asks an injected TokenProvider for one; every host builds its own broker using whatever is native to it (VS Code's built-in account system, Electron's safeStorage, or a small shared CLI/MCP helper); and a token never once touches the NogginDocument, the noggin-rpc wire, or a log line.

The six rules

Everything below is a consequence of these. If you only read one section, read this one.

  1. The engine never knows about auth. No token, header, or OAuth concept enters @noggin/engine's core (noggin-api.mjs/.d.mts). Only a provider module and its own type declarations know anything about credentials — the same way file.mjs knows about fs and http.mjs knows about fetch, while core knows about neither.
  2. A provider never logs anyone in. It calls token = await opts.auth.getToken(). If that's undefined, it throws auth-required and stops. Signing in is a 100% host concern, never a provider concern.
  3. Every host builds its own broker and holds its own tokens, using whatever is native to that host. VS Code delegates entirely to vscode.authentication and holds nothing itself. Desktop encrypts via Electron's built-in safeStorage. CLI/MCP share one small Node helper that prefers delegating to gh auth token, then falls back to noggin's own device-flow login.
  4. Tokens never cross a boundary that didn't already exist for a good reason. Never inside a NogginDocument, never on the noggin-rpc wire, never in a log line or an error's message/data. Split-process hosts wire brokers in at server-construction time, in-process — not as a request/response payload.
  5. Interactive vs. silent is one boolean, not two designs. TokenProvider.getToken({ interactive }) — silent by default (never pops UI); interactive: true only from a place the user just asked for it. Every host implements both branches however is native to it; nothing upstream needs to know which host it's running in.
  6. Exactly one small, additive wire-protocol change. A new provider.reauth method — because reusing provider.open for "refresh this exact session" would risk silently switching the user to a different repo. Everything else rides through channels (NogginErrorCode, the existing free-form opts bag) that already exist.

Where auth lives in the existing layering

Nothing about the engine/provider/host split changes. Auth just picks a side for each new concept — and it's almost always the host's side.

engine — untouched provider — defines the seam host — does the actual work never crosses here
@noggin/engine core — noggin-api.mjs / .d.mts
NogginDocument, verbs, NogginError, the provider registry interface. Gains three new additive error codes (auth-required, auth-forbidden, auth-rate-limited) and nothing else. Zero imports of fetch, OAuth, or any credential concept.
↓ a provider is host/network-shaped code that plugs into the registry
engine/providers/github.mjs (+ github.d.mts)
New provider module, same shape as http.mjs. Defines and exports the TokenProvider type here, not in core — it's a provider-local contract, like FileFilter is host-shaped and lives beside the code that needs it. Calls opts.auth.getToken(); never touches storage, browsers, or OAuth endpoints itself.
↓ opts.auth is constructed and injected by whichever process owns the engine
Host process — CLI, MCP, desktop main, or VS Code extension host
Owns exactly one concrete TokenProvider implementation per identity: drives the interactive sign-in UX, decides where bytes are stored, decides when to refresh or invalidate. This is where 90% of the actual new code in this proposal lives — see the per-host matrix.
The no-fly zone. A token is constructed inside the host process and consumed inside the (also host-side) provider — both live in the same trust boundary already, so the hop between them is a plain in-process function call, never a message. It must never be written into a NogginDocument, returned from noggin.describe(), included in a noggin-rpc request or response, or interpolated into a NogginError.message/data field. Everywhere a real implementation is tempted to "just log the request for debugging," that's exactly where this rule gets broken in practice — treat Authorization header redaction as a lint-checked convention, not a one-time reminder.

Why CLI/MCP and desktop/VS Code look asymmetric: they genuinely are. Single-process hosts have no interactive-UI substrate at all (CLI takes flags, MCP takes tool JSON over stdio, sometimes with no TTY); split-process hosts already have a whole native-dialog layer (HostServices) to build on. Forcing one shared "auth module" across both families would be the actual design smell — the honest shared surface is the small TokenProvider type, not a shared implementation.

Token flows

Four sequences: the two ways a user ever signs in, and the two ways an already-open noggin reacts to a credential problem.

1 · First sign-in — desktop, CLI, or MCP (device flow)

GitHub's own recommended flow for headless/native apps — no redirect URI, no local webserver, no CSRF state parameter needed anywhere.
sequenceDiagram
    autonumber
    participant U as User
    participant H as Host process - desktop main, CLI, or MCP
    participant B as Broker (TokenProvider)
    participant GH as github.com

    U->>H: Sign in with GitHub
    H->>B: getToken(interactive: true)
    B->>GH: POST /login/device/code
    GH-->>B: device_code, user_code, verification_uri
    B-->>H: user_code + verification_uri
    H-->>U: Open github.com/login/device, enter ABCD-1234
    U->>GH: enters code, approves requested scopes
    loop every interval seconds, until approved or expired
        B->>GH: POST /login/oauth/access_token
        GH-->>B: authorization_pending
    end
    GH-->>B: access_token, refresh_token, expires_at
    B->>B: encrypt and persist (session-only if no OS keychain)
    B-->>H: token
    H-->>U: Signed in successfully
Phishing mitigation, not a footnote. Device flow's known weakness is code-relay phishing: an attacker starts their own flow and tricks a victim into entering the attacker's code on the real GitHub site. Two concrete mitigations, both must-fix, not optional: (1) noggin only ever displays a code the host itself just requested — it never has a field asking a user to paste in someone else's code; (2) every place the code is shown says, verbatim, "only enter this if you just started this sign-in in noggin." If the team later confirms GitHub Apps support a secret-less PKCE token exchange, a loopback-redirect flow (RFC 8252) is a strictly stronger option for desktop specifically, since desktop can safely assume the browser and the app share a machine — CLI over SSH cannot make that assumption, so device flow stays required there regardless.

2 · First sign-in — VS Code (fully delegated)

The easy host. Zero custom storage, zero custom UI, zero device-flow code.

    U->>Ext: open or create a github noggin
    Ext->>VSA: getSession('github', scopes, createIfNone: true)

    note over Ext,VSA: noggin persists nothing here - VS Code owns the whole session

3 · Steady state — a provider making an authenticated call

Same code path on every host. The provider never knows which broker it's holding.
sequenceDiagram
    autonumber
    participant P as github provider (engine)
    participant B as Broker (TokenProvider)
    participant GH as api.github.com

    P->>B: getToken()  (silent)
    B-->>P: token
    P->>GH: GET /repos/owner/repo/issues, Authorization: Bearer token
    alt 200 OK
        GH-->>P: issues array
        P-->>P: map into NogginDocument
    else 401 Unauthorized
        GH-->>P: 401
        P->>B: invalidate()
        P-->>P: throw NogginError(code: auth-required)
    else 403 secondary rate limit
        GH-->>P: 403, x-ratelimit-reset
        P-->>P: throw NogginError(code: auth-rate-limited)
    end

4 · Reactive re-auth on an already-open noggin

The one place a genuinely new RPC method earns its keep — see wire impact for why.

    RPC-->>UI: snapshot

What actually changes on the wire

The strongest property of this design: almost none of it touches @noggin/rpc. Everything a token-handling flow needs either stays entirely host-side, or rides a channel that already existed for an unrelated reason.

Unchanged

  • NogginDocument, every verb's semantics, the JSON schema
  • Every existing RpcProtocol method and type
  • HostServices — still exactly the same 7 methods
  • ProviderFlows — still exactly create / pickToOpen / listInstances / describe
  • file://, https:///http://, memory://, localstorage:// — completely untouched

Additive, backward-compatible

  • 3 new NogginErrorCode values: auth-required, auth-forbidden, auth-rate-limited — provider-agnostic, reusable by ADO / Planner / anything future, not GitHub-specific
  • 1 new optional RPC method: provider.reauth(scheme, location)
  • 1 new convention on the already free-form opts bag: opts.accountKey (default 'default') so a host can hold more than one identity per provider
  • 1 new provider module: engine/providers/github.mjs (+ .d.mts exporting TokenProvider)
  • 1 new host-only constructor option: createNogginRpcServer({ authBrokers })

Never, by design

  • A token, header, or OAuth concept inside @noggin/engine's core
  • A token on the noggin-rpc wire, in either direction
  • A token inside a NogginDocument, a describe() string, a log line, or NogginError.data
  • A required change to any existing provider, host, or UI component that doesn't care about GitHub
Why provider.reauth and not just reusing provider.open? provider.open(scheme) is documented as "pick a possibly-new location" — reusing it to mean "refresh the credential for the exact noggin that's already open" risks a user's own recents list silently steering them into a different repo mid-reauth. A location-scoped method removes the ambiguity for one small, reviewable addition.

Storage & at-rest protection

HostMechanismAt-rest protectionFallback behaviour
VS Code vscode.authentication session (VS Code core) Whatever VS Code's own platform-native secret storage does — noggin never implements this n/a — nothing is ever persisted by the extension itself
Desktop Electron safeStorage.encryptString → blob under userData OS-keychain-backed (Keychain / DPAPI / libsecret), via Electron's built-in API — no third-party native module isEncryptionAvailable() false → session-only, never written to disk in the clear; user sees a one-time notice that re-auth will be needed next launch
CLI / MCP tiered: gh auth token → optional OS-keychain module → chmod 600 JSON file tier-dependent; the file fallback is permissions-only, matching ~/.npmrc / ~/.aws/credentials conventions the file tier is a deliberate, accepted-industry-norm last resort, not a silent downgrade — print a one-time stderr warning the first time it's used
MCP has a real limitation here, not a bug to paper over. "MCP reads whatever the CLI already stored" assumes the MCP server runs as the same OS user/profile as the interactive CLI session — true for the common case (an agent host spawning noggin-mcp on your own laptop), false for a containerized or remote-sandboxed agent. In that case an OS-keychain-backed credential simply can't be decrypted from a different account. Document this explicitly; if a shared-credential path is needed across a container boundary, it must be an explicitly-mounted file path the deployment configures — never an environment variable or CLI argument, both of which leak via process listings and shell history.

The actual new surface, in code

The one shared contract

// engine/providers/github.d.mts — NOT in noggin-api.d.mts.
// A provider-local type, same precedent as http.mjs already using fetch
// without core engine knowing what fetch is.

export interface TokenProvider {
  /**
   * Resolve a currently valid token.
   *   interactive: false (default) — never shows UI; undefined if signed out.
   *   interactive: true  — may run an interactive sign-in, only called from
   *   a place the user just asked for it (a provider's create/open flow, an
   *   explicit "Sign in" command, or provider.reauth).
   */
  getToken(opts?: { interactive?: boolean }): Promise<string | undefined>;

  /** Called after a 401-like failure. Clears any cache so the next
   *  getToken() call re-prompts instead of returning a stale token. */
  invalidate(): Promise<void>;
}

export interface GithubProviderOptions {
  auth: TokenProvider;
  /** Which stored identity to use when a host holds more than one
   *  (personal vs. work, github.com vs. an enterprise host). */
  accountKey?: string;
}

Three new engine error codes (additive)

// noggin-api.d.mts — appended to the existing NogginErrorCode union
  | 'auth-required'      // no valid credential; host must run an interactive sign-in
  | 'auth-forbidden'     // credential valid but lacks permission/scope for this resource
  | 'auth-rate-limited'  // upstream rate limit; data.retryAfterSeconds if known

One new RPC method (additive)

// @noggin/rpc protocol.ts — appended to RpcProtocol
'provider.reauth': { request: ProviderReauthRequest; response: ProviderReauthResponse };

interface ProviderReauthRequest  { readonly scheme: string; readonly location: string; }
interface ProviderReauthResponse { readonly ok: boolean; }

One new host-only server option

// Purely host-side construction. Never serialized; never sent anywhere.
createNogginRpcServer({
  transport,
  hostServices,
  providerFlows,
  authBrokers: (scheme: string, accountKey: string) => TokenProvider | undefined, // NEW
});

Risk register — what the adversarial reviews changed

Two independent reviews (security-focused, architecture-focused) were run against an earlier draft of this design. Every must-fix below already changed the design above; this section is the paper trail, not a backlog.

Security review

must-fix

Device flow is phishable via relayed codes (real attacks exist against M365/Okta-style device flows).

Explicit "only enter this if you just started this sign-in" UX copy everywhere a code is shown; loopback+PKCE documented as a stronger opt-in for desktop.

must-fix

Electron safeStorage can be unavailable (Linux with no keyring daemon) and some versions degrade silently.

Explicit isEncryptionAvailable() check; fail to session-only, never a plaintext-on-disk fallback.

must-fix

One broker per URI scheme silently mixes two accounts (personal vs. work, or github.com vs. an enterprise host) open at once.

Brokers resolved by (scheme, accountKey); accountKey rides the pre-existing free-form opts bag — no protocol change.

must-fix

Two near-simultaneous 401s could spawn two competing device-code prompts.

Broker coalesces concurrent interactive getToken() calls into a single in-flight promise.

must-fix

GitHub App user tokens expire in hours; no refresh path means constant re-login.

Stored schema includes refreshToken + expiresAt; the silent path refreshes proactively before expiry.

must-fix

A shared/malicious noggin could smuggle an internal host as a "GHES base URL" (SSRF via a trusted process holding a real bearer token).

v1 ships github.com only; any future enterprise-host value must come from trusted host-side settings, never from document/location content, and must be allow-listed.

must-fix

owner/repo interpolated into a fetch URL with no stated validation.

Strict charset validation before building the URL; error data fields are an explicit allow-list, never a raw upstream passthrough.

should-fix

MCP assumes the same OS-user identity as the interactive CLI session — false in a container/sandbox.

Documented as a real limitation; a containerized MCP needs an explicitly-mounted shared path, never env var/argv passthrough.

should-fix

Silent downgrade to the permissions-only file fallback is exactly what infostealer tooling already targets.

One-time, explicit stderr warning the first time the file-fallback tier is used — never a silent downgrade.

should-fix

Several github:// noggins sharing one identity could independently trip GitHub's secondary rate limit.

The broker holds shared per-token rate-limit bookkeeping; the provider checks before firing a request.

should-fix

No way to pre-authenticate before ever opening a github noggin.

Every host gets an explicit, discoverable "Sign in to GitHub" affordance, independent of any open noggin.

Architecture review

must-fix

Reusing provider.open for "refresh this exact session" risks silently switching the user to a different repo picked from a QuickPick default.

Added one small, additive, location-scoped provider.reauth(scheme, location) method — the design's own original hedge ("if this feels awkward") was already an admission it was needed now, not later.

must-fix

Folding "ensure signed in" into ProviderFlows.create/ pickToOpen wouldn't survive the next two providers — Planner's session model is VS Code's own account picker; ADO's is a cached multi-step QuickPick. Each is shaped differently.

Generalized as TokenProvider.getToken({ interactive }), callable from anywhere host-side. ProviderFlows itself needs no interface change at all.

must-fix, self-resolving

"No protocol change" seemed to break down once desktop's device-flow needed an in-app "waiting for you to authorize…" progress UI that none of the 7 HostServices methods model.

Resolved as a side effect of the phishing-mitigation fix above: desktop's UX is "open a browser, react when it's done" — no in-app polling-status UI, so no new HostServices method is needed on any host.

should-fix

Putting TokenProvider in core engine would erode "the engine has zero host/network knowledge," and it wouldn't stay a one-off once ADO/Planner land.

Defined in the github provider's own .d.mts instead — the same precedent as provider-local, host-shaped types already living beside their provider, not in core.

worth noting

CLI/MCP and split-process hosts get genuinely asymmetric implementations, not one shared "auth module."

Accepted as a deliberate tradeoff — the two host families have real capability differences; forcing a shared implementation would be the actual smell. See the per-host matrix.

Phasing & open decisions

DecisionRecommendation
App registrationRegister noggin as a GitHub App, not a legacy OAuth App — users pick exactly which repos to install it on, tokens are short-lived, and revocation is per-installation rather than account-wide. This is GitHub's own current guidance, not just this document's preference.
Scopes, v1Read-only issues access. Ship the provider read-only first (mirrors how http.mjs already ships read-only-by-design) — closing/editing issues is a v2 question about mapping done/edit/note onto GitHub mutations, deliberately out of scope for the auth design itself.
Multi-accountSupported from day one via accountKey — retrofitting it later would be a breaking change to however hosts persist locations (recents lists, workspace files).
GitHub Enterprise ServerNot in v1. When it lands, the base host must come from trusted host-side settings the user explicitly configures, never from a noggin location string, and must be validated against an allow-list — see the SSRF item in the risk register.
Device flow vs. loopback+PKCE for desktopShip device flow first (identical code path to CLI/MCP, well-documented phishing mitigations). Revisit loopback+PKCE only if the team confirms GitHub Apps support a secret-less token exchange for installed apps — shipping a client_secret inside a distributed Electron binary is its own, separate bad practice, so this isn't a free upgrade.
