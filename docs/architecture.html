noggin — architecture northstar

All split-process hosts share one transport-agnostic noggin-rpc protocol. Engine + providers always run host-side; UI is a thin remote client. Click any box for details.

Two host families, one protocol

Hosts split cleanly: single-process hosts (CLI, MCP, agents) call the engine directly. Split-process hosts (desktop, VS Code, future web) place engine + providers in the trusted process and reach them through noggin-rpc. The transport differs per host; the protocol does not.

The shift: previously we considered putting the engine inside the webview/renderer. That works for the file provider (small interface, known shape) but doesn't scale to providers that need network access, OS keychain, or external API auth. Putting engine + providers host-side and using noggin-rpc for everything generalises cleanly to every imaginable provider.
One word, on purpose: a provider is the single unit that teaches noggin how to deal with one kind of place where noggins live — both the data side (read/write/watch) and the UX side (create/open/list/describe). We deliberately don't separate "backend" from "provider" in the design vocabulary; they're the same author shipping the same integration.
The cost we accept: verbs are now async over IPC even in desktop. We pay this back with an optimistic-update layer in the webview/renderer that applies the mutation locally before the host confirms. Done correctly, the UI feels in-process even though it isn't.

noggin-rpc — the protocol

Transport-agnostic, symmetric, request/response + streaming. Designed so a postMessage host and an electron-ipc host and a future WebSocket host all expose exactly the same service.

Symmetry is the point. A web host that has WebSocket but no postMessage just swaps the transport adapter. A future plugin runtime (codex-cli, claude code, etc.) that wants to render the UI in-shell pulls in the rpc client + ui components. Every "thing the UI does" is one of these methods.

CLI host

Bypasses noggin-rpc. Engine + providers all in one process. Reference behaviour every other host's verbs match.

No noggin-rpc here. Single process means in-process verb calls. This host stays simple and fast. (MCP server has the same shape; the JSON-RPC layer there is the agent-protocol, not noggin-rpc.)

Desktop host (Electron) — split-process with noggin-rpc

Engine + providers + host services live in the main process. The renderer is a pure UI shell, talking to main via electron-ipc-wrapped noggin-rpc. Optimistic UI keeps gestures sync-feeling despite the round-trip.

Architectural simplification, not regression. Yes, we lose the desktop's beautiful in-process model — but we regain it through optimistic UI, and we get cross-host symmetry as the prize. The renderer's contextIsolation: true + nodeIntegration: false tightening is a bonus security win.

VS Code host — same shape, different transport

Identical model to desktop. Engine + providers + host services run in the extension host. Webview talks noggin-rpc over postMessage. Providers can call any VS Code API — including network, secrets, workspace.fs for virtual filesystems.

Symmetric with desktop. Same engine code, same provider code, same React code, same noggin-rpc method names. Only the transport adapter differs (electron-ipc vs postMessage). The same workspace provider can be authored once and registered in either host where it makes sense (probably VS-Code-only, but trivial to also wire into desktop if we ever want).

Providers — host-side UX

A provider answers: how do I create / open / list / describe a kind of noggin. They run host-side so they can call any platform API. The UI invokes them through noggin-rpc and pays no attention to where the work happens.

What this buys: a new provider is purely host-side code. It cannot break the UI's invariants because everything it does is mediated by the noggin-rpc method surface. The UI doesn't need to know whether a provider talks to disk, a REST API, or a USB device.

Agents — the fourth host

Two transports for two scenarios: MCP for cross-tool standardisation, vscode.lm tools for in-VS-Code chat. Both reuse the host's engine instance.

The agent integration story is simpler with this architecture: in both transports, the agent hits the same engine the user is editing. ChangeEvents flow through the same noggin-rpc bus the UI subscribes to. No special cases anywhere.
