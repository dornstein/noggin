The UI always talks to a Noggin interface. Sometimes that's the real engine (direct call); sometimes it's a RemoteNoggin that forwards verbs over IPC. The UI can't tell the difference.

Split process — who talks to whom

Solid grey arrows are ordinary function calls inside one process. Coloured dashed arrows are messages that cross the process boundary — that's the IPC. Each colour is a separate channel.

desktop (Electron)engine in main · UI sandboxed in renderer
noggin-rpc
The main path. UI verbs go right→left; results and live change events come back left→right. Everything the UI does flows here — file dialogs now ride this too, as host services.
noggin-host-services-rpc
The reverse arc — main is the client here. When a verb needs UI main can't draw, it forwards the call to the renderer's HostServicesReactImpl and reads the answer back. The dashed box marks the split: one HostServices, two halves.

Two channels — no more shell. The native menu is built-in roles + Help links that stay wholly in main; the app's own actions (new/open/close, sidebar, details layout, providers, about) live in the renderer's controls and keyboard accelerators.

extension (VS Code)engine in the extension host · UI in the webview

Only one channel. The extension host already owns real VS Code dialogs (vscode.window.showInputBox, showQuickPick, showOpenDialog), so every HostServices call is answered right there — nothing has to call back into the webview. Electron's main can't draw an input box, which is the one reason the desktop still needs the extra noggin-host-services-rpc channel; its native menu stays wholly in main.
