# Docs site

The source for [https://dornstein.github.io/noggin/](https://dornstein.github.io/noggin/).

Built and deployed by [`.github/workflows/pages.yml`](../../.github/workflows/pages.yml)
on every push to `main`.

## Layout

```
docs/site/
  build.mjs              entry point; orchestrates the whole build
  template.mjs           shared HTML chrome (sidebar nav, footer)
  markdown.mjs           tiny markdown → HTML renderer (no external deps)
  assets/style.css       site stylesheet
  pages/*.md             hand-written content pages
  generators/            scripts that produce HTML from live source
    cli.mjs                runs `noggin help`
    api.mjs                parses engine/**/*.d.mts for @public / @internal
    mcp.mjs                imports mcp/noggin-mcp.mjs TOOLS array
    schema.mjs             renders noggin.schema.json
```

The legacy [`scripts/build-demo-html.mjs`](../../scripts/build-demo-html.mjs)
is invoked by `build.mjs` to produce the `/demo/` page.

## Build locally

```bash
node docs/site/build.mjs --out docs/site/dist
```

Open `docs/site/dist/index.html` in a browser. The output is in
`.gitignore`.

## Run locally (build + serve)

For iterative work, use the zero-dep dev server. It builds the site,
serves it over HTTP, and rebuilds on source changes:

```bash
node docs/site/serve.mjs --watch
# → serving docs/site/dist at http://localhost:8080/
```

Flags: `--port 4000`, `--no-build` (skip the initial build),
`--watch` (rebuild on edits to `pages/`, `generators/`, `playground/`,
`assets/`, `cli/`, or `scripts/`).

From VS Code: **Tasks: Run Task → `docs: serve site`** (background
task, runs the same command).

Why HTTP and not `file://`? Relative links like `../assets/style.css`
work fine either way, but the in-browser playground keys its
`localStorage` per origin — opening via `file://` and via
`http://localhost` give you two different stores. Use the server for
anything beyond a quick spot-check.

## Adding a page

1. Drop a markdown file under `pages/` with frontmatter:

   ```markdown
   ---
   title: Some page
   slug: "some-page/"
   ---
   ```

   `slug` is the URL slug (always trailing-slashed; `""` for root).
   The page renders to `<out>/<slug>index.html`.

2. Add an entry to `NAV` in [`template.mjs`](./template.mjs) so it
   appears in the sidebar.

3. `node docs/site/build.mjs` and check the result.

## Dynamic content

The site never hand-mirrors information that's already in the
source — every dynamic generator reads from canonical files in the
repo so docs and code can't drift:

| Page | Source of truth |
|---|---|
| `/cli/` | `noggin help` output (rebuilt CLI on every build) |
| `/api/**/` | `engine/noggin-api.d.mts`, every `engine/providers/*.d.mts`, `engine/serializers/*.d.mts` — split into per-symbol pages by [`generators/api.mjs`](./generators/api.mjs) |
| `/mcp/` | `mcp/noggin-mcp.mjs` (the `TOOLS` array imported directly) |
| `/schema/` | `noggin.schema.json` |
| `/demo/` | `scripts/build-demo-html.mjs` (runs real CLI scenarios) |

If you change one of those sources, the next CI build picks it up
automatically. No manual sync needed — with one caveat: the API
tree is split into pages by an explicit manifest, so a **new**
public engine export needs a one-line addition to that manifest.
See "API-reference authoring" below.

## API-reference authoring

The `/api/**/` tree is generated by
[`generators/api.mjs`](./generators/api.mjs) from TypeDoc output. It
runs TypeDoc once, indexes every emitted symbol section by name,
then routes each symbol to a page via the `PAGES` manifest at the
top of the generator. The manifest is the whole layer of policy:

```js
const PAGES = [
  {
    slug:    'api/errors/noggin-error/',       // URL slug
    title:   'NogginError',                    // <title> + h1
    intro:   '# `NogginError`\n\n…prose…',     // hand-written intro
    symbols: [{ module: 'noggin-api', name: 'NogginError' }],
  },
  // …
];
```

- `slug` — must match a nav entry in [`template.mjs`](./template.mjs).
- `intro` — hand-authored markdown. Owns the page's H1 and any
  editorial framing. TypeDoc's own H3 for a single-symbol page is
  suppressed to avoid repeating the name.
- `symbols` — one or more `{module, name}` refs. On multi-symbol
  pages each symbol becomes an H2 boundary; on single-symbol pages
  the intro provides the framing.

### Drift audit

`buildApiPages()` fails the whole docs build if:

- A public engine symbol isn't referenced by any page ("orphan").
- The manifest names a symbol TypeDoc didn't emit ("ghost").

The failure message names the specific symbol and how to fix it.
The audit is the invariant that keeps this system honest — new
engine exports can never silently disappear from the docs.

### Adding a public engine export

1. Add the symbol to `engine/noggin-api.d.mts` (or the appropriate
   `.d.mts`) with a TSDoc description and a release tag
   (`@public`, `@experimental`, `@internal`).
2. Add a `PAGES` entry in [`generators/api.mjs`](./generators/api.mjs).
   Related pure functions (e.g. path helpers) cluster on a shared
   page; substantial types get their own.
3. If it's under a new group, also create
   `pages/api/<group>/index.md` (the group overview) and add a
   subgroup block in `template.mjs`'s `NAV`.
4. Run `node docs/site/build.mjs --out docs/site/dist` locally.
   The audit tells you if anything's missing.

### Adding a new provider

Provider-specific symbols (`openXNoggin`, `xProvider`, `OpenXOptions`)
belong on their narrative provider page, not per-symbol under
`/api/`. Wiring:

1. Add the `.d.mts` to `entryPoints` in [`typedoc.json`](./typedoc.json).
2. Add the module name to `MODULE_FILES` in `generators/api.mjs`.
3. Add a `MODULE_FALLBACK_SLUGS` entry pointing at the narrative
   page (`'providers/x': 'providers/x/'`). This makes TypeDoc
   cross-refs into the provider module resolve to the narrative
   page rather than 404.
4. Write the narrative page at `pages/providers/x.md`.
5. Add the nav entry under the `Providers` top-level group in
   `template.mjs`.

The audit will not flag provider-module symbols as orphans (they
default to "narrative page owns them"). If you actually want a
provider symbol to have its own per-symbol API page, add its name
to `EXPECTED_PROVIDER_ORPHANS` and route it via a `PAGES` entry.

### Renaming or removing an export

The audit will flag the stale manifest entry as a ghost. Update
or remove the entry in `PAGES`, then grep the narrative markdown
(`pages/**/*.md`) for the old name in case a link references it.

### Cross-references

TypeDoc emits `[Foo](#foo)` and `[Foo](../other.md#bar)` links
inside its markdown. `generators/api.mjs` rewrites both:

- Hash refs whose base name maps to a different page become
  relative URLs into that page.
- Inter-module `.md` refs are resolved through the same anchor map,
  falling back to `MODULE_FALLBACK_SLUGS` for provider modules.
- A post-render HTML pass repairs any numeric suffix drift
  (`#foo-4` → `#foo-3` when `-3` exists on the target page).

You should never need to touch this code when adding an export —
the audit + rewriter handle everything as long as the manifest is
up to date.

## Markdown subset supported

The hand-rolled renderer in [`markdown.mjs`](./markdown.mjs) covers
the common cases: headings, paragraphs, fenced code blocks with
language hints, inline code, bold/italic, links, images, ordered
and unordered lists, pipe tables, blockquotes, horizontal rules, and
inline HTML passthrough (so we can drop `<div class="card-grid">`
etc. directly in markdown). If you need something more exotic, just
write HTML inline — the renderer leaves it alone.
